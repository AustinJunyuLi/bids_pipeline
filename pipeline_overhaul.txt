в—Ҹ Good question. The deterministic stages serve three roles relative to
  extraction quality вҖ” let me break down what each could do better.

  1. Pre-Extraction (Shaping What the LLM Sees)

  This is the highest leverage. Better preprocessing = better LLM input = fewer
  errors to catch.

  Block-level metadata enrichment. Right now blocks carry block_id, start_line,
  end_line, raw_text, clean_text, is_heading. The preprocessing could
  deterministically annotate each block with:
  - Date mentions extracted via regex (you already have DATE_FRAGMENT_RE in
  evidence scanning вҖ” apply it per-block)
  - Entity mentions вҖ” simple pattern matching for "Party A/B/C", company names
  from the seed, known advisor patterns
  - Evidence density score вҖ” count of evidence items overlapping this block
  - Temporal phase hint вҖ” "initiation" / "bidding" / "closing" based on position
   + content signals

  This metadata costs zero LLM tokens and directly enables the prompt-side
  improvements (temporal phase tags, evidence density markers, block importance
  hints).

  Evidence scanning expansion. The current 5 cue families (dated_action,
  financial_term, actor_identification, process_signal, outcome_fact) are good.
  But the coverage stage later finds gaps that the scanner missed вҖ” those gap
  patterns could feed back into the scanner's regex vocabulary. Essentially:
  mine your coverage failures for new patterns.

  2. Post-Extraction (Catching LLM Errors Better)

  Temporal consistency check. Not currently a gate. Events should be
  chronologically ordered relative to their evidence positions in the filing. If
   event A's evidence appears at line 1500 but event B's appears at line 1200,
  and A's date is after B's, that's fine. But if A's evidence is at line 1200,
  B's at line 1500, and A is dated AFTER B вҖ” the LLM may have confused which
  passage maps to which event. This is a pure deterministic check.

  Attention decay diagnostics. The verify stage knows WHERE quote resolution
  fails (which blocks). Aggregate that: if all failures cluster in blocks
  B080-B140, that's direct evidence of mid-document attention decay. This
  diagnostic could automatically trigger the targeted recovery strategy вҖ” and
  it's purely deterministic analysis of existing verification findings.

  Per-actor coverage. Coverage currently checks event-type coverage against
  evidence cues. It doesn't check whether every actor who signed an NDA
  eventually appears in a proposal, drop, or outcome event. An actor who signs
  an NDA then vanishes is either a missed drop or a missed event вҖ” detectable
  deterministically from the actor roster + event timeline.

  Cross-event logic validation. Deterministic rules like:
  - No proposal event after an executed event (in the same cycle)
  - No NDA event after a drop for the same actor (without a restart)
  - Every bidder who reaches "executed" must have at least one prior proposal
  - Round announcements should precede round deadlines

  These are domain invariants that the LLM might violate. Currently no gate
  checks them.

  3. Enrichment (Already Strong, One Gap)

  Timeline reconstruction. enrich-core does rounds, bid classifications, cycles,
   and formal boundary. But it doesn't produce a validated chronological
  timeline вҖ” an ordered sequence of (date, actor, event_type) tuples that could
  be compared against the raw chronology for sanity. This would catch date
  resolution errors that verify doesn't flag (correct quote, wrong date).
