● Good questions. Let me think through both.

  1. Batch processing

  Right now the pipeline is serial per deal — /deal-agent <slug> runs one deal
  end-to-end. For 400 deals, you need to think about three things:

  What's parallelizable vs what's not:
  - Deterministic stages (fetch, preprocess, compose-prompts, canonicalize,
  check, verify, coverage, gates, enrich-core) — all independent per deal,
  trivially parallelizable
  - DuckDB db-load — file-level lock means sequential writes, but each load
  takes <1 second so this isn't the bottleneck
  - LLM calls (extract-deal, verify-extraction, enrich-deal) — these are the
  bottleneck. Each deal takes minutes of LLM time. 400 deals serially = days

  Simplest batch approach (no code changes):
  A shell loop calling /deal-agent <slug> for each seed. Serial, slow, but works
   today. For 400 deals at ~10 min per deal, that's ~67 hours.

  Better batch approach (moderate code changes):
  - Split the pipeline into a deterministic pre-stage (fetch through
  compose-prompts) that runs as a Python batch loop
  - Fan out LLM extraction calls in parallel (N concurrent deals)
  - Converge back to deterministic post-stages (canonicalize through db-export)
  as another batch loop
  - The DuckDB load becomes a single batch pass at the end

  The real constraint is LLM throughput. With API calls you can parallelize
  across deals. With local-agent skills, you're limited to one agent at a time
  (unless you spawn multiple Claude Code sessions).

  This connects directly to your second question.

  2. Python-orchestrated API calls vs local-agent skills

  What you have now: Three LLM stages are local-agent skills — Claude reads the
  SKILL.md, reads the prompt packets, calls itself, writes artifacts. This works
   but has limitations:

  - No parallelism — one agent, one deal at a time
  - No retry logic — if the agent crashes mid-extraction, you restart manually
  - No token tracking — you don't know cost per deal
  - No rate limit management — can't throttle across 400 concurrent requests
  - Context window waste — the agent loads SKILL.md, CLAUDE.md, project context
  every time, burning tokens on orchestration overhead

  What Python API calls would give you:
  - Parallel extraction — fire 10-20 deals concurrently with asyncio
  - Structured outputs — use Pydantic response models directly (no parsing agent
   text)
  - Automatic retries with exponential backoff
  - Token counting and cost tracking per deal
  - Rate limit handling via the SDK
  - Deterministic prompt assembly (you already have compose-prompts building the
   exact packets)

  What you'd lose:
  - Agent flexibility — the LLM can't "think about" what to do, it just fills a
  structured response
  - Repair loops — verify-extraction's interactive repair would need to become a
   programmatic retry
  - The "no Python LLM wrapper" constraint from CLAUDE.md — this was a v1 design
   choice, not a physics constraint

  My honest take: For 400 deals, you almost certainly need Python API calls. The
   local-agent approach was right for v1 (getting the extraction contract
  correct with 9 deals). But scaling to 400 deals with agent-driven extraction
  is impractical — too slow, too fragile, too expensive on orchestration tokens.

  The good news: you already built the hard part. compose-prompts generates the
  exact prompt packets. The structured output schema is defined in Pydantic
  models. The deterministic gates validate the output. You'd essentially be
  replacing the middle step — instead of "agent reads prompt packet and writes
  JSON," it's "Python sends prompt packet to API and parses structured
  response."

  The migration path would be:
  1. Add a skill-pipeline extract --deal <slug> Python stage that calls the API
  with the composed prompt packets
  2. Add a skill-pipeline extract-verify --deal <slug> that does programmatic
  repair (re-send failed quotes)
  3. Add a skill-pipeline enrich-interpret --deal <slug> for the interpretive
  enrichment call
  4. Keep the local-agent skills as a manual/debug fallback
  5. Build a batch runner: skill-pipeline batch --deals all --parallel 10

  That's essentially your v2 milestone scope.

❯ but local agnets can spawn subagents right?                                   

● Yes — Claude Code can spawn subagents via the Agent tool, and each subagent
  gets its own context window. So in theory you could:

  Main agent (orchestrator)
    ├── Agent: deal-agent imprivata    (parallel)
    ├── Agent: deal-agent saks         (parallel)
    ├── Agent: deal-agent medivation   (parallel)

  1. DuckDB file lock — all deals write to the same data/pipeline.duckdb.

  account rate limit. 10 parallel extractions may throttle.
  3. Context cost — each subagent loads CLAUDE.md, SKILL.md, filing text, prompt
   packets. That's ~50-100k tokens of overhead per deal before any extraction
  work. At 400 deals, that's 20-40M tokens just in orchestration context.
  4. No retry/resume — if a subagent crashes at deal #37, you have to figure out
   which deals completed and re-run the rest manually.
  5. No token tracking — you can't see cost per deal or total spend.

  So yes, local agents CAN parallelize via subagents, and for 9-20 deals it's
  viable. For 400 deals, the overhead and fragility push you toward Python API
  calls where you control the concurrency, retries, and cost tracking directly.

  The sweet spot might be hybrid: keep the local-agent path for
  development/debugging (run one deal interactively), add the Python API path
  for batch production (run 400 deals with proper concurrency and monitoring).
