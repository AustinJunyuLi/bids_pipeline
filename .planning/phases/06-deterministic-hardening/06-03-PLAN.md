<objective>
Add bounded DuckDB lock retry at the shared database entry point so `db-export`
can survive transient file-lock contention immediately after `db-load` without
masking unrelated connection/configuration errors.

Purpose: HARD-05 is a connection-boundary fix, not a db-export-specific hack.
The retry belongs in `open_pipeline_db()` because all deterministic DuckDB
callers already route through that helper.

Output: a lock-aware retry wrapper in `skill_pipeline/db_schema.py` and
regression tests covering eventual success, exhausted retries, and non-lock
errors.
</objective>
