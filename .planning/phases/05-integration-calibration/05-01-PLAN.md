For each `(event_id, classification)` in `det_data["bid_classifications"]`, build enrichment row with `bid_label=classification["label"]`, `bid_rule_applied=classification.get("rule_applied")`, `bid_basis=classification.get("basis")`, and `dropout_label=None`, `dropout_basis=None`.

For each `(event_id, classification)` in `interp_data.get("dropout_classifications", {})`, update the matching enrichment row's `dropout_label=classification["label"]` and `dropout_basis=classification.get("basis")`.
         3. executemany INSERT all enrichment rows.
         Note: `deterministic_enrichment.json` does NOT contain `dropout_classifications`. That key only exists in the optional interpretive `enrichment.json`. The load must handle both tiers gracefully.
       - `_load_cycles(con, deal_slug, data)` - from enrichment dict `data["cycles"]`
       - `_load_rounds(con, deal_slug, data)` - from enrichment dict `data["rounds"]`
    8. Return 0.

    **Step 1: Create `skill_pipeline/db_export.py`.**
    Module structure:
    ```python
    def run_db_export(deal_slug: str, *, project_root: Path = PROJECT_ROOT) -> int:
    ```
    Implementation:
    1. `paths = build_skill_paths(deal_slug, project_root=project_root)`
    2. `db_path = paths.database_path`
    3. If not `db_path.exists()`: raise `FileNotFoundError(f"Pipeline database not found: {db_path}")`
    4. `con = open_pipeline_db(db_path, read_only=True)`
    5. Verify deal exists: `SELECT COUNT(*) FROM events WHERE deal_slug = ?`. If 0: raise `ValueError(f"No events found for deal '{deal_slug}' in database")`
    6. Query all data needed:
       - `actors = _query_actors(con, deal_slug)` -> list of dicts
       - `events = _query_events(con, deal_slug)` -> list of dicts
       - `enrichment = _query_enrichment(con, deal_slug)` -> dict keyed by event_id
       - `cycles = _query_cycles(con, deal_slug)` -> list of dicts
       - `rounds = _query_rounds(con, deal_slug)` -> list of dicts
    7. Build CSV rows using the export-csv SKILL.md spec:
       - `_assign_bidder_ids(events)` implementing the 5-step algorithm:
         a. Sort events by `date_sort` (nulls last), then by type priority (process markers first, NDA, proposal, drop, round, outcomes)
         b. Find first NDA index
         c. Pre-NDA events get fractional IDs at `1/(N+1)` increments rounded to 1 decimal
         d. Post-NDA events get integer IDs starting at 1
       - `_format_event_row(event, actors_by_id, enrichment, bidder_id, seen_actor_types)` builds 14-column row:
         - `bidderID`: the assigned ID
         - `note`: map event_type to note value per SKILL.md table. For `drop`, look up enrichment `dropout_label` from DuckDB enrichment table (default "Drop" if NULL). Dropout labels like "DropAtInf" come from the interpretive enrichment overlay loaded by db-load.
         - `bidder`: actor display_name for bidder events, NA for process events
         - `type`: bidder type code (most-specific-first priority from SKILL.md). Only on first row per actor, NA on subsequent rows. Track via `seen_actor_types` set.
         - `bid_type`: from enrichment bid_classifications ("Formal", "Informal", or NA). "Uncertain" -> NA.
         - `val`: terms.per_share or NA
         - `range`: "low-high" or "val-val" for point bids, or NA
         - `date_r`: formatted per 3 rules (exact=`YYYY-MM-DD 00:00:00`, imprecise=approximated, unknown=NA)
         - `date_p`: exact=same as date_r, imprecise=NA, unknown=NA
         - `cash`: "1" if consideration_type=="cash", else NA
         - `c1`, `c2`, `c3`: reserved for interpretive enrichment fields (initiation_judgment, advisory_verification, count_review). Populate from DuckDB enrichment table if columns exist and are non-NULL; otherwise NA.
         - `review_flags`: populate from DuckDB enrichment table if available; otherwise NA.
         Note: The quality of the CSV export depends on which enrichment tier was loaded by db-load. With deterministic-only enrichment, dropout labels default to "Drop" and c1/c2/c3 are NA. With the interpretive overlay, the CSV matches the full export-csv SKILL.md spec. The exit criterion "matches or improves on JSON-based export" requires running /enrich-deal before db-load for full parity.
    8. Write deal header block:
       - Load seed entry for deal metadata
       - Write 2-row header: column names then values (TargetName, Events count, Acquirer, DateAnnounced, URL)
    9. Write event rows as CSV after the header.
    10. `ensure_output_directories(paths)`, write to `paths.deal_events_path`.
    11. `con.close()`, return 0.
