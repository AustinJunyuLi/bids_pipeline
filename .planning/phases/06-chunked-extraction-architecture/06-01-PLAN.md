### Extraction Method

#### Chunk Construction

Split `chronology_blocks.jsonl` into sequential chunks of ~10-12 blocks each (~3-4K tokens per chunk). Rules:
- Every block appears in exactly one primary chunk.
- Adjacent chunks share 1-2 blocks of overlap to catch boundary-spanning events.
- Heading blocks (is_heading=true) are never chunk boundaries -- include the heading with the following narrative block.
- Chunk numbering: chunk_1, chunk_2, ..., chunk_N.

For each chunk, scope evidence items from `evidence_items.jsonl`: include only items whose `start_line`..`end_line` range intersects the chunk's block range.

#### Chunk Extraction (Sequential)

Process chunks in order: chunk_1 first, then chunk_2, etc. Each chunk receives:
1. Its block window (the ~10-12 blocks assigned to this chunk, rendered as numbered text)
2. Scoped evidence items (only items intersecting this chunk's line range)
3. Actor roster from prior chunks (empty for chunk_1, accumulated roster for subsequent chunks)

Each chunk extracts actors AND events together (not separated). The chunk prompt must:
- Extract all actors visible in this chunk's blocks. If a party is not already on the roster from prior chunks, mint a new actor with a new actor_id and evidence_refs from this chunk.
- Extract all events visible in this chunk's blocks, following the 20-type taxonomy. Assign temporary per-chunk event_ids (e.g., `c1_evt_001`, `c2_evt_003`).
- Extract count_assertions visible in this chunk's blocks.
- Do NOT set `after_final_round_announcement` or `after_final_round_deadline` -- these require cross-chunk history and are set by consolidation.

Roster carry-forward: After each chunk completes, merge its new actors into the running roster. The next chunk receives the full roster so far.

Chunk outputs are transient debug artifacts written to `data/skill/<slug>/extract/chunks/` (one JSON file per chunk). They are NOT consumed by deterministic stages.

#### Consolidation Pass (Replaces Pass 2)

After all chunks are processed, run one consolidation pass. The consolidation pass receives:
1. Final actor roster (accumulated across all chunks)
2. All chunk event drafts (structured JSON from every chunk)
3. All count_assertions (merged across chunks)
4. Evidence gap signals (block ranges with no extracted events, if any)

The consolidation pass performs:
1. **Actor dedup by canonical_name**: Group actors by (canonical_name.strip().upper(), role, bidder_kind). Pick survivor with most evidence_refs. Merge evidence_refs from duplicates into survivor. Log merges.
2. **Event dedup for overlap zones**: For events extracted in overlap blocks (blocks that appear in two adjacent chunks), dedup by (event_type, date, actor_ids). Keep the event with the richer summary. Merge evidence_refs.
3. **20-type taxonomy sweep**: For each of the 20 event types, confirm at least one event was extracted OR write `NOT FOUND -- [specific filing-based reason]` in coverage_notes.
4. **Count reconciliation**: Compare count_assertions against extracted actor/event counts. Flag mismatches for downstream.
5. **Targeted re-reads**: For each event type with `NOT FOUND`, re-read the 3-5 most relevant blocks for that type. Extract any missing events.
6. **Temporal signals**: Set `after_final_round_announcement` and `after_final_round_deadline` on proposal events by scanning the consolidated event timeline for prior round events.
7. **Global event_id assignment**: Replace temporary per-chunk event_ids (c1_evt_001, etc.) with stable global event_ids (evt_001, evt_002, ...) in chronological order. Update all actor_ids references and invited_actor_ids.

#### Output

Consolidation writes the final `actors_raw.json` and `events_raw.json` to `data/skill/<slug>/extract/`. These use the same schema as today. No changes to artifact paths, Pydantic models, or downstream stage contracts.
